# Plan 05-02: DB-Level Sum-to-100 Trigger + Post-Mutation Recalculation Queue

## Objective
Create a PostgreSQL `BEFORE INSERT/UPDATE/DELETE` trigger on `grade_items` that validates `SUM(max_score_basis_points) = assessment.total_basis_points` at the database level to prevent data-entry errors where teachers overshoot the assessment total. Additionally, build an Asynq queue task for cascading score recalculation triggered automatically on grade item create/delete and manually on max_score edits.

## Requirements
PREC-03, PREC-04

## Success Criteria
- [ ] `BEFORE INSERT/UPDATE/DELETE` trigger on `grade_items` validates sum of all grade item `max_score_basis_points` equals the parent assessment's `total_basis_points`
- [ ] Trigger error message includes assessment ID and current/expected sums for debugging
- [ ] New Asynq task type `score:recalculate` with payload containing `assessment_id`, `session_id`, `school_id`
- [ ] Asynq task handler calls existing `RollupAssessmentScore` (or a bulk variant) for all students in the assessment
- [ ] Auto-enqueue on grade item create/delete (structural changes)
- [ ] Manual-enqueue only on max_score edits (teacher-initiated via button or API)
- [ ] `RecalcNeeded` flag on Assessment model cleared after successful recalculation
- [ ] Frontend displays existing `RecalcNeeded` badge when assessment needs rollup (via `RecalcWarningBadge`)
- [ ] `go build ./...`, `go vet ./...`, existing tests pass

## Tasks

### 1. Create the sum-to-100 DB trigger

- [ ] **1.1** Add a new migration entry in `backend/internal/database/migrations/school/school.go`:
  - `CREATE OR REPLACE FUNCTION check_grade_items_sum()` that:
    - For INSERT/UPDATE: reads `NEW.assessment_id`, sums all `max_score_basis_points` for that assessment (including `NEW` value), fetches `assessment.total_basis_points`
    - For DELETE: reads `OLD.assessment_id`, sums all remaining grade items, fetches `assessment.total_basis_points`
    - If sum != total, raises exception: `'Grade items sum BP %d does not match assessment total BP %d (assessment %d)'`
  - `DROP TRIGGER IF EXISTS trg_check_grade_items_sum ON grade_items`
  - `CREATE CONSTRAINT TRIGGER trg_check_grade_items_sum AFTER INSERT OR UPDATE OR DELETE ON grade_items FOR EACH ROW EXECUTE FUNCTION check_grade_items_sum()`
    - Use AFTER + CONSTRAINT (deferrable initially immediate) so the trigger sees the final state after the mutation
  - `DROP FUNCTION IF EXISTS check_grade_items_sum` in rollback
- [ ] **1.2** Test with raw SQL: insert grade items summing to total (pass), then insert an item that overshoots (fail with clear error).

### 3. Create the Asynq task handler

- [ ] **3.1** Create `backend/internal/queue/handlers/score_recalculate_handler.go`:
  - Deserialise `ScoreRecalculatePayload` from `t.Payload()`
  - Get tenant DB via schema resolver (school_id → schema name)
  - Fetch all student IDs enrolled in the assessment (via level/curriculum/assessment relationships)
  - For each student, call `ScoreService.RollupAssessmentScore` (or a new batch variant that skips student-level validation)
  - Clear `Assessment.RecalcNeeded` after successful rollup
  - Log errors per-student but don't stop the batch (`logger.Warnf`)
- [ ] **3.2** Add a new method `RollupAssessmentForAllStudents` on `ScoreService` (or modify existing rollup loop) that accepts assessment ID and session ID, iterates all (student, subject) combos, and runs rollup for each.
