<objective>
Convert tenant-scoped handler methods and service methods across all modules to route per-school data through the school's dedicated tenant database.

**Wave 1 scope**: academic (28 handlers), timetable (7), exam (12), result (2), score (5) = 54 handlers total

**Pattern** (established in user module — reference implementation):

**handler.go** — each handler method extracts repos and passes TenantDB to service:
```go
// Add to handler imports:
"gorm.io/gorm"
"github.com/playbits/schoolcare-v2/internal/middleware"

// Inside each handler method:
repos := middleware.GetTenantRepos(c)
var tenantDB *gorm.DB
if repos != nil {
    tenantDB = repos.TenantDB()
}
result, err := h.service.Method(c.Request.Context(), tenantDB, schoolID, req)
```

**service.go** — each service method accepts optional tenantDB and uses it:
```go
func (s *Service) Method(ctx context.Context, tenantDB *gorm.DB, schoolID uint, req *Req) (*Model, error) {
    db := s.repo.GetDB()
    if tenantDB != nil {
        db = tenantDB
    }
    // Create temp repo scoped to tenant DB for write operations
    tempRepo := NewRepository(db)
    tempRepo.Create(ctx, model)
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| handler→service | Tenant DB connection is resolved per-request in the handler and passed to the service. The service trusts whatever DB connection it receives. |
| middleware→handler | TenantDBResolver middleware runs before all authGroup handlers. If it fails (no tenant DB available), it passes through silently — handler gets nil tenantDB and falls back to core DB. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | S (Spoofing) | Tenant DB resolution | mitigate | TenantDBResolver relies on school_id from JWT (already validated by JWTAuth middleware). School ID cannot be spoofed because JWT is signed. |
| T-06-02 | T (Tampering) | tenantDB param in service | mitigate | tenantDB is read-only in the handler (never modified). Service uses it only to create temp repos. The original core-DB repos are never modified, ensuring thread safety. |
| T-06-03 | I (Info Disclosure) | Fallback to core DB | accept | When tenant DB is nil (school not yet provisioned), operations fall back to core DB with school_id filter. Same security model as before — school_id scoping prevents cross-school data access. |
| T-06-04 | D (DoS) | Temp repo creation | accept | Creating a new AcademicRepository struct is an O(1) heap allocation per request. No performance impact at scale. |
</threat_model>
