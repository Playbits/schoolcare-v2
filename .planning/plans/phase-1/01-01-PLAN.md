  <files>
    backend/internal/database/migrations/migrator.go
  </files>
  <behavior>
    - Test 1: Run() obtains a unique advisory lock for each schema before applying migrations
    - Test 2: Run() releases the lock at transaction commit/rollback (xact lock = auto-release)
    - Test 3: Concurrent Run() with same schema name serializes (second caller waits)
    - Test 4: Run() skips lock and returns error if schema name is empty
    - Test 5: Already-applied migrations are skipped regardless of lock state
  </behavior>
  <action>
    Modify `ReusableMigrator.Run()` to wrap execution in a database transaction with `pg_advisory_xact_lock`.

    **Lock key scheme**: Use a 64-bit bigint hash of the tracking table name. Derive via FNV-1a or a simple polynomial hash of the table name string. This gives each migration tracking table a unique lock ID — core DB and each tenant schema get separate lock namespaces.

    **Implementation**:
    1. At the start of `Run()`, begin a GORM transaction with `tx.Exec("SELECT pg_advisory_xact_lock($1)", lockKey)`.
    2. Inside the transaction: create tracking table (IF NOT EXISTS), iterate migrations with the existing `Already-applied` check, run each pending migration, record in tracking table.
    3. If any migration fails, the transaction rolls back, auto-releasing the lock. Return the error.
    4. On success, commit releases the lock.

    **Key changes**:
    - `Run()` now takes an optional schema identifier string parameter (for tenant migrations). Core migrations use "core" as the identifier.
    - The lock key is computed as `hash(identifier + tableName)` — ensures core and tenant migrations use different lock IDs even with same table name.
    - Keep `Run()` signature backward-compatible: add a variadic `opts ...RunOption` or add a new `RunWithSchema(schemaName string) error` method that wraps `Run()`.

    **Edge cases**:
    - Empty schema name → lock key uses table name only (safe for core migrations that don't pass a schema).
    - Migration tracking table `CreateTable` DDL is inside the transaction — if a second caller holds the lock, the DDL serializes correctly.

    **Why pg_advisory_xact_lock** (not `pg_try_advisory_lock` or session-level lock):
    - Transaction-level lock auto-releases on commit or rollback — no cleanup needed.
    - Blocks until lock is available (serializes, doesn't fail).
    - Works with PgBouncer transaction mode because the lock is held within a single transaction on a single connection.
    - GORM's `Transaction()` callback ensures the lock is acquired and released on the same connection.

    **Lock hash function** (append to migrator.go or a small helper):
    ```go
    // migrationLockKey derives a unique 64-bit advisory lock key from
    // the tracking table name and an optional schema identifier.
    // Hash is FNV-1a 64-bit: fast, deterministic, collision-resistant for this use case.
    func migrationLockKey(tableName, schemaName string) int64 {
        h := fnv.New64a()
        h.Write([]byte(tableName))
        h.Write([]byte(":"))
        h.Write([]byte(schemaName))
        return int64(h.Sum64())
    }
    ```
