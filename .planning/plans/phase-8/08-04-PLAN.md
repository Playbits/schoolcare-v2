# 08-04: Cache Warming — Provisioning + Background Refresh

## Objective
Enable proactive cache warming for tenant resolution: call `WarmCacheByUUID()` on provisioning completion, and add a periodic background refresh for active schools.

## Background
`TenantResolutionService.WarmCache()` and `WarmCacheByUUID()` are fully implemented but never called. Every first request to a school pays a cold-start penalty (DB lookup for tenant context). This plan connects the warming triggers.

## Tasks

### Task 1: Warm cache on provisioning
1. In `backend/internal/database/tenant/provisioning.go`, after school is marked as "active" (step 6 of `ProvisionSchool`), call `WarmCacheByUUID()`.
2. Problem: `ProvisioningService` doesn't have access to `TenantResolutionService` or the school UUID. Options:
   - **Option A**: Pass `TenantResolutionService` to `ProvisioningService` constructor
   - **Option B**: Pass `schoolUUID` parameter to `ProvisionSchool` and have the caller handle warming
   - **Option C**: Make calling code (the provisioning task handler or school service) handle warming
3. **Decision**: Option C — the caller handles warming. Modify the school creation flow:
   - In `school/service.go` `Create()` method, after provisioning succeeds, call `tenantResolutionService.WarmCacheByUUID(ctx, school.UUID)`
   - Inject `TenantResolutionService` into `SchoolService`
4. Update `SchoolService` constructor in `setup.go`.

### Task 2: Add background cache refresh
1. Create a new job in `scheduler/jobs.go` or `backend/internal/scheduler/cache_warm_job.go`:
   - `CacheWarmJob` — enumerates active schools (where `database_status = 'active' AND schema_name IS NOT NULL`)
   - For each school, calls `TenantResolutionService.ResolveTenant(ctx, schoolID)` to refresh the cache
   - Logs count of refreshed caches
   - Default schedule: every 30 minutes (`*/30 * * * *`)
   - Configurable via `CRON_CACHE_WARM` env var in SchedulerConfig
2. Inject `TenantResolutionService` into the job
3. Register with CronRunner in setup.go

### Task 3: Handle nil Redis gracefully
1. Both warming paths should check if Redis is available first
2. `TenantResolutionService.WarmCache()`/`WarmCacheByUUID()` already calls `ResolveTenant()` which handles nil Redis gracefully (just skips caching). So this is already handled.
