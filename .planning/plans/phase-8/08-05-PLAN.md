### Task 2: Fix inconsistent annotations across all existing modules
Standardize patterns across ALL ~30 modules with existing annotations:

1. **Route prefix consistency**: Ensure all `@Router` paths use `/api/v2/...` prefix (some use relative paths like `/audit-logs`)
2. **Add @Failure annotations**: Every endpoint should document possible error responses (400, 401, 403, 404, 500)
3. **Add pagination @Param annotations**: Every list endpoint should document `page` and `limit` query params
4. **Standardize response model references**: Use `response.APIResponse` consistently
5. **Fix @Security annotations**: Ensure all authenticated endpoints have `@Security BearerAuth`
6. **Add @Produce json**: Where missing
7. **Fix @Success response descriptions**: Replace generic "OK" with meaningful descriptions

### Task 3: Add request body examples
For write operations (POST, PUT, PATCH), add `@Example` values:
- User creation: `{ "email": "john@example.com", "password": "...", "role": "student" }`
- School creation: `{ "name": "Example School", "plan": "free" }`
- Session creation: `{ "name": "2025/2026", "year": "2025/2026", "term": "1" }`
- Score save: `{ "student_id": 1, "grade_item_id": 1, "score": 75.5 }`
- etc.

Not every endpoint needs an example — focus on the most commonly used write operations.

### Task 4: Run swag init and fix warnings
1. Run `swag init -g cmd/server/main.go -o cmd/server/docs`
2. Fix any warnings (missing descriptions, inconsistent types, etc.)
3. Verify generated `swagger.yaml` parses correctly
4. Add `make swagger` target to Makefile (or update existing one)
5. Add step to run `swag init` in CI or document as post-build step

### Task 5: Add CI / documentation for swagger regeneration
1. Update Makefile swagger target to use proper paths
2. Add small check to CI (optional — verify generated docs match handler annotations)
